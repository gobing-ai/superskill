# F008 тАФ SQLite data store

## What

SQLite store under the `.superskill/` data root resolved by `content/paths.ts` (F007): `<cwd>/.superskill/evaluations.db` when `<cwd>/.superskill/` exists, else `~/.superskill/evaluations.db`; `--project`/`projectRoot` forces project-local (ADR-013). **Data access goes through `@gobing-ai/ts-db` (ADR-014), never `bun:sqlite` directly.** Tables are authored once via `defineTable` (single source of truth for table + zod schemas + DDL); `store/evaluations.ts` and `store/proposals.ts` are thin `EntityDao` subclasses. Provides CRUD for evaluation records and evolution proposals.

## Why

The `evaluate --save` and `evolve` operations require persistent storage for longitudinal quality tracking. This is the self-evolution data foundation. ADR-007 mandates preferring `@gobing-ai/ts-*` over raw/external data access, and ts-db is the project's data-access facade тАФ hand-rolling `bun:sqlite` with literal DDL would violate ADR-007 and forfeit boundary validation, migration tracking, and D1 portability the facade already provides (ADR-014). Centralizing the path rule in F007 ensures `evaluate --save` and `evolve` always open the **same** DB.

## Change

### Dependencies

- Declare in `apps/cli/package.json`: `@gobing-ai/ts-db`, `drizzle-orm`, `drizzle-zod`, `zod` (all already resolved transitively; ADR-014). No `bun:sqlite` import anywhere in `store/`.

### `store/schema.ts` (table definitions тАФ single source of truth)

- Author both tables with `defineTable` from `@gobing-ai/ts-db/schema`:
  - `evaluations` тАФ **append-only** (`appendOnlyColumns`: `created_at` only, no `updated_at`):
    `id` (PK, autoincrement integer), `content_type`, `content_name`, `target_agent`, `operation`, `aggregate` (real), `dimensions` (text/JSON), `file_hash` (text, nullable), + `appendOnlyColumns`.
  - `proposals` тАФ mutable status (`standardColumns`: `created_at` + `updated_at`):
    `id` (PK), `content_type`, `content_name`, `baseline_id` (nullable int), `proposal_json` (text/JSON), `status` (text, default `'draft'`), `applied_at` (text, nullable), `verify_id` (nullable int), + `standardColumns`.
- Export the derived `.table`, `.insertSchema`, `.selectSchema` for each тАФ DDL is generated by `defineTable`, **never hand-written**.

### `store/db.ts`

- Export `openStore(opts?: { projectRoot?: string }): Promise<DbAdapter>` тАФ calls `createDbAdapter({ driver: 'bun-sqlite', url: getDBPath(opts) })`, then `applyMigrations(adapter, { тАж })` to create/track the two tables. Idempotent across opens.
- Path resolution is delegated to `content/paths.ts` (F007) тАФ `store/db.ts` re-exports `getDBPath` but does **not** re-implement the data-root rule.
- Tests pass `{ projectRoot }` (a tmpdir) or `url: ':memory:'` for isolation тАФ no real `~/.superskill/` writes.

### `store/evaluations.ts` (EntityDao subclass)

- Export `class EvaluationDao extends EntityDao<тАж>` constructed with `(adapter, evaluations.table, idColumn, { insertSchema: evaluations.insertSchema })`.
- `insertEvaluation(record: EvaluationInput): Promise<number>` тАФ `create(record)`; returns the new row id. Validates via the derived zod `insertSchema` at the boundary.
- `getEvaluations(contentType, contentName): Promise<Evaluation[]>` тАФ `list({ where: and(eq content_type, eq content_name), orderBy: [{ col: created_at, dir: 'desc' }] })` via the predicate query spec (no raw SQL).
- `getLatestEvaluation(contentType, contentName): Promise<Evaluation | null>` тАФ same predicate, `limit: 1`.
- `EvaluationInput`: `{ content_type, content_name, target_agent, operation: 'evaluate' | 'refine' | 'evolve', aggregate, dimensions: Record<string, { score: number, note: string }>, file_hash? }`
  - `operation` is **always supplied by the caller** тАФ `evaluate`тЖТ`'evaluate'`, `refine`тЖТ`'refine'`, evolve verifyтЖТ`'evolve'`. The store never defaults it.
  - `target_agent` is non-null; callers pass `'claude'` when `--target` is omitted.
  - `file_hash` comes from `hashContent()` (F007).
- `dimensions` is serialized as JSON on write, parsed on read (DAO maps the text column).

### `store/proposals.ts` (EntityDao subclass)

- Export `class ProposalDao extends EntityDao<тАж>` with the `proposals` table.
- `insertProposal(record: ProposalInput): Promise<number>` тАФ `create({ ...record, status: 'draft' })`; returns id.
- `updateProposalStatus(id, status, opts?): Promise<void>` тАФ `update(id, { status, applied_at?, verify_id? })`.
- `getProposals(contentType, contentName): Promise<Proposal[]>` тАФ `list` predicate ordered by `created_at desc`.
- `getPendingProposals(): Promise<Proposal[]>` тАФ `list({ where: eq(status, 'draft') })`.
- `ProposalInput`: `{ content_type, content_name, baseline_id?, proposal_json: object }`
- `Proposal`: `ProposalInput & { id, status, applied_at, verify_id, created_at, updated_at }`
- `proposal_json` is serialized as JSON on write, parsed on read.

> **Note (async).** ts-db DAO methods are `async` (the adapter supports bun-sqlite and D1). All store call sites in F011/F013 therefore `await` store calls.

## Acceptance

```
import { openStore } from './store/db';
import { EvaluationDao } from './store/evaluations';

const adapter = await openStore({ projectRoot: tmpDir });
const dao = new EvaluationDao(adapter);
await dao.insertEvaluation({
  content_type: 'skill',
  content_name: 'test',
  target_agent: 'claude',
  operation: 'evaluate',
  aggregate: 0.82,
  dimensions: { completeness: { score: 0.85, note: 'ok' } },
});
const rows = await dao.getEvaluations('skill', 'test');
// rows.length === 1 тАФ no bun:sqlite import in store/, no hand-written DDL/SQL
```
